# Code Style + Cursor Prompts

> Универсальные правила понятного кода и промпты для Cursor.
> Применять в **каждом** проекте на 100%.

---

## Одна страница на стену

1. **Понятные имена** — без сокращений, по смыслу
2. **Одна функция — одно дело** — < 30 строк
3. **Без `any`** — типы или `unknown` + guard
4. **Без магических чисел/строк** — константы
5. **Ошибки видимы** — никогда `catch {}`
6. **Структура по фичам** — `features/<name>/{ui,lib,model,hooks}`
7. **Линтер всегда зелёный** — CI блокирует
8. **Атомарные коммиты** — повелительное наклонение
9. **Секреты в `.env`** — никогда в git
10. **README — точка входа** — клон → запуск за 15 минут

## Таблица 1. Главные правила (100%)

| # | Правило | Что делать | Зачем |
|---|---------|------------|-------|
| 1 | **Имена объясняют намерение** | Существительное для данных, глагол для действий. Без сокращений. Boolean — `isLoading`, `hasError` | Код читают в 10× чаще, чем пишут |
| 2 | **Одна сущность — одна ответственность** | Файл/функция делает одно. Если в названии «и» — разделить | Легче тестировать и менять |
| 3 | **Маленькие функции и файлы** | Функция ≤ 30 строк, файл ≤ 150–200 строк, вложенность ≤ 3 | Большие файлы = свалка |
| 4 | **Без магических значений** | Числа и строки → константы (`MAX_RETRIES`, `API_BASE`) | Меняешь в одном месте |
| 5 | **Строгая типизация** | Никаких `any`. `unknown` + guard. Явные return types | Компилятор ловит баги до прода |
| 6 | **Чистые функции** | Утилиты без side effects. IO/state изолированы | Легко тестировать |
| 7 | **Ошибки видимы** | Никаких пустых `catch {}`. Лог или UI | Тихие баги = ад в проде |
| 8 | **Валидация на границе** | API, форма, парсинг — валидируем. Внутри — доверяем типам | Защита от мусора |
| 9 | **DRY с умом** | Дублирование 3+ → выносим. 1–2 — оставляем | Преждевременная абстракция дороже |
| 10 | **Комментарии — про "почему"** | Удалить «// увеличить i». Оставить «// legacy API» | «Что» видно из кода |
| 11 | **Структура по фичам** | `features/order/{ui,lib,model}` | Удалить фичу = удалить папку |
| 12 | **Чёткие границы слоёв** | UI ↔ логика ↔ данные. UI не лезет в БД | Можно менять одно, не ломая другое |
| 13 | **Иммутабельность** | `const`, не мутировать args/state | Меньше гонок и багов |
| 14 | **Early return** | Сначала guard clauses, потом логика | Меньше вложенности |
| 15 | **Линтер обязателен** | ESLint + Prettier. CI блокирует merge | Один стиль = ноль споров |
| 16 | **Атомарные коммиты** | Один коммит = одно изменение. «Add», «Fix» | Легко revert и читать историю |
| 17 | **Секреты не в коде** | `.env` + `.env.example` | Секрет в git = компрометация |
| 18 | **README актуальный** | Что, как запустить, как тестить | Новый дев за 15 минут |
| 19 | **Тесты для критичного** | Валидация, бизнес-правила, утилиты | Безопасный рефакторинг |
| 20 | **Читаемость > краткости** | Длинное имя > хитрый one-liner | Код для людей |

## Таблица 2. Anti-patterns — никогда

| Чего избегать | Почему | Как правильно |
|---------------|--------|---------------|
| `any`, `@ts-ignore` | Теряется типобезопасность | `unknown` + guard |
| `catch (e) {}` | Тихие баги | Лог + показать/пробросить |
| Магические числа | Никто не помнит смысл | `const MAX = 7` |
| Функции 200+ строк | Не помещается в голове | Разделить |
| Глубокая вложенность | Глаза разбегаются | Early return |
| Закомментированный код | Гниёт | Удалить — git помнит |
| Mutating arguments | Непредсказуемость | Новое значение |
| Секреты в git | Утечка | `.env` |
| `utils.ts` на 1000 строк | Свалка | По доменам |
| Преждевременная оптимизация | Усложняет | Сначала измерь |

## Таблица 4. Именование

| Что | Стиль | Пример | Анти-пример |
|-----|-------|--------|-------------|
| Переменная | camelCase | `userEmail` | `e`, `data1` |
| Boolean | is/has/can | `isActive` | `flag` |

| Компонент | PascalCase | `JournalScreen` | `journalscreen` |

# Cursor Prompts

## AGENTS.md (положить в корень каждого проекта)

Скопируйте блок ниже в файл `AGENTS.md`:

```markdown
# Правила для AI-ассистента

Ты — аккуратный senior-разработчик. Пишешь код, который легко читать, поддерживать и ревьюить.

## Обязательные правила

1. **Имена**: понятные, без сокращений. Boolean — is/has/can. Файлы kebab-case, компоненты PascalCase, хуки useXxx.
2. **Один файл — одна ответственность**. Файл < 150 строк, функция < 30 строк.
3. **Без any**. unknown + type guard. Явные return types.
4. **Без магических значений** → константы.
5. **Ошибки видимы**. Никаких пустых catch {}.
6. **Валидация на границе** (API, форма).
7. **Чистые функции** для утилит.
8. **Структура по фичам**: features/<name>/{ui,lib,model,hooks}.
9. **Иммутабельность**: const, не мутировать state/args.
10. **Early return** вместо вложенных if/else.
11. **Комментарии — про "почему"**, не "что".
12. **DRY**: дублирование 3+ → выносить.
13. **Импорты**: внешние → @/ → относительные.
14. **Секреты в .env**, в репо .env.example.
15. **Читаемость > краткости**.

## Запрещено

- any, @ts-ignore (кроме generated)
- Пустой catch {}
- Магические числа
- Закомментированный код
- Mutating arguments
- Преждевременная абстракция
- Файлы в node_modules, .next, dist, generated

## Стиль работы

- Сначала читай существующий код. Подстраивайся.
- Не переписывай то, что не просили. Минимальный диф.
- Используй установленные зависимости.
- После правок: линтер + билд.
- Коммиты атомарные, в повелительном наклонении. Без Co-authored-by.

## Чек перед завершением

- [ ] Линтер зелёный
- [ ] Типы проверены
- [ ] Нет console.log, debugger
- [ ] Нет закомментированного кода
- [ ] README обновлён (если менялось поведение)

## Когда не уверен — спрашивай, не выдумывай.
```

## Промпт: новая фича

```
Реализуй [описание] по правилам из AGENTS.md.

Контекст:
- [бизнес-задача]
- [куда: features/<name>]
- [связанные файлы]

Требования:
- features/<name>/{ui,lib,model,hooks}
- Типы отдельно, логика в хуках
- Валидация на сервере + UX на клиенте
- Без any, без магических значений

Если неясно — спроси до начала.
```

---

## Промпт: рефакторинг

```
Отрефактори [файл/папка] по AGENTS.md, не меняя поведение.

Цели:
1. Файлы < 150 строк
2. Функции < 30 строк
3. Вынести типы, утилиты, логику
4. Убрать дубли (3+)
5. Улучшить имена

Не трогать: публичные API, generated/, node_modules/
После: линтер + билд.
```

---

## Промпт: багфикс

```
Исправь баг: [описание + шаги].

1. Найди корень (не симптом)
2. Покажи причину перед фиксом
3. Минимальный дифф
4. Проверь смежное

Не меняй неотносящееся.
```

## Промпт: старт проекта

```
Создай проект [название] на [стек].

Настрой:
- AGENTS.md
- ESLint + Prettier + TypeScript strict
- .env.example, .gitignore
- README (1 команда запуска)
- src/features/, src/lib/, src/app/

Покажи план перед созданием.
```

Прочитай AGENTS.md и подтверди правила.

## Промпт: ревью

```
Проверь [файл/PR] по AGENTS.md.

Формат:
1. **Критично** (надо исправить)
2. **Желательно** (стоит улучшить)
3. **Хорошо** (что отлично)

Не предлагай переписывать всё.
```

## Промпт: одиночная задача (без AGENTS.md)

```
Сделай [задача].

Обязательно:
- TypeScript строго, без any
- Файлы < 150 строк, функции < 30
- Структура по фичам: features/<name>/{ui,lib,model}
- Чистые функции для утилит
- Ошибки обрабатывать, не глотать
- Имена понятные, без сокращений
- Линтер должен пройти

Сначала покажи план — потом делай.
```
